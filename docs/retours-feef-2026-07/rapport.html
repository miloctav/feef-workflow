Le détail

A

Un filtre invisible masque tout l'historique

Les tableaux d'audits appliquent un filtre par défaut qui exclut les audits terminés.

const initialFilters = urlStatus
  ? { status: [...] }
  : { inProgress: true }   // exclut COMPLETED, REFUSED_*

Le même composant est monté sans barre de filtres dans l'onglet Audits d'une fiche entité et dans « Mes audits » côté entreprise. L'utilisateur ne voit donc ni le filtre, ni le moyen de le retirer.

Ce que ça produit

  • Biscuiterie Bouvard : 7 audits en base, 5 terminés donc masqués. Seuls 2 s'affichent.
  • LSDH : 8 audits terminés sur 10, invisibles.
  • Maison Collet : son unique audit est terminé. L'onglet paraît entièrement vide, ce qui a fait croire à un import raté.
À vérifier en 10 secondes : ouvrir la fiche Maison Collet et ajouter ?status=COMPLETED à l'URL. Si l'audit apparaît, l'import est confirmé bon.
B

Comment le statut est déterminé à l'import Ecocert

Le fichier source contient 1126 lignes. Trois filtres successifs s'appliquent.

  1. Filtre contrat. Seules les lignes Contracted sont importées : 1060 sur 1126. Les 66 Not contracted sont ignorées.
  2. Filtre suiveuse. Si l'entité est une suiveuse, ses audits ne sont pas importés : le label est porté par le groupe ou le siège. Règle validée par Ecocert.
  3. Traduction du statut. La colonne AUDIT_STATUT d'Ecocert est traduite en statut de workflow FEEF.
switch (statutEcocert) {
  case '4-certification finished'        → TERMINÉ
  case '3-inspection finished - pool'    → EN ATTENTE AVIS OE
  case '3-certification in progress'     → EN ATTENTE DÉCISION FEEF
  case '3-certification order accepted'  → EN ATTENTE DÉCISION FEEF
  case '2-inspection in progress'        → EN ATTENTE RAPPORT (si date de fin passée)
  default                                → EN PLANIFICATION
}
si EN PLANIFICATION et qu'une date planifiée existe → PLANIFIÉ

La dernière règle est celle qui pose problème : tout statut non reconnu devient « planifié » dès qu'une date existe, sans aucun contrôle sur l'année.

C

Les suiveuses de LSDH sont invisibles

« LSDH n'a pas ses suiveuses mais L'Abeille est bien rattachée au groupe LSDH »

La donnée est correcte. LSDH a bien ses 4 suiveuses en base : L'ABEILLE, LES CRUDETTES, JUS DE FRUITS D'ALSACE, FRUITS ET SAVEURS. Une seule entité LSDH, aucun doublon.

La cause

LSDH est enregistrée en type = Entreprise, pas Groupe. Le seed ne promeut une mère en « Groupe » que si elle est créée de toutes pièces. LSDH existait déjà dans le fichier source avec son propre SIRET, elle est donc restée « Entreprise ».

Or l'interface discrimine sur ce type :

type === GROUPE      → affiche les entités filles      ✓
type === ENTREPRISE  → affiche le groupe parent (null) ✗

Les 4 suiveuses sont bien chargées par l'API, mais jamais lues. L'export, lui, lit la relation dans l'autre sens (chaque fille affiche son parent), d'où le « pas rattachées dans le workflow, mais ok sur l'extract ».

Décision prise : LSDH sera passée en « Groupe » manuellement via l'interface. Le cas d'une entreprise maître portant des suiveuses est jugé incohérent et ne sera pas supporté.
D

Pourquoi les graphiques sont blancs

Les titres s'affichent, le reste est vide. La console dit :

Uncaught TypeError: Cannot read properties of null (reading 'save')
    at Bt._drawDataset  →  Bt.draw  →  ul._update

Chart.js dessine une instance déjà détruite. Le scénario :

  1. Le composant graphique monte avant son parent. Les données ne sont pas encore chargées, mais le garde de sécurité ne bloque pas : un « graphique fantôme » vide est créé et son animation démarre.
  2. Le parent charge les données.
  3. Le graphique se redessine, ce qui détruit le graphique fantôme.
  4. L'animation du fantôme tourne toujours. Au frame suivant elle dessine sur un canvas détruit et lève l'exception.
  5. L'exception casse la boucle de rendu, partagée entre tous les graphiques. Les deux restent blancs.

Cela explique que le bug ne se voie pas en local : le chargement y est assez rapide pour arriver avant le montage.

Ce n'est pas un problème de données. 28 audits tombent dans la fenêtre des 12 prochains mois, et les 852 audits terminés se répartissent sur 2022 à 2026 (110 / 211 / 218 / 228 / 85). Les deux graphiques ont largement de quoi s'afficher.
E

L'email de Sarah, et le mode développement

La production tourne avec NUXT_DEV_MODE=true. Cause connue, à corriger séparément. Trois conséquences en cascade :

  1. Le mail est détourné. En mode dev, tout destinataire est remplacé par maxime@miloctav.fr, en dur dans la configuration. L'email d'activation de Sarah est parti chez Maxime.
  2. Le compte est créé déjà actif, avec le mot de passe password.
  3. Le renvoi est impossible : l'endpoint refuse un compte déjà actif.

Trois faiblesses qui subsisteront même après le passage en production

  • L'erreur d'envoi est avalée. L'API retourne « compte créé » même si le mail a échoué. L'interface n'affiche jamais cette information.
  • Aucun bouton « renvoyer l'email ». L'endpoint existe, il n'est exposé nulle part. C'est la réponse à son « quel est le process ? » : aujourd'hui, aucun recours.
  • Le lien peut être invalide si nginx ne propage pas les en-têtes X-Forwarded-* : il pointerait sur localhost:3000.

À valider par ailleurs : le domaine expéditeur notifications.monpmeplus.fr est-il vérifié dans Resend (SPF/DKIM) ?

Bloc Décision

  • Ordre des cartes : codé en dur, une ligne à déplacer.
  • Trou réel : le statut « validation du plan d'action » n'a aucune carte, alors qu'il est compté dans le bandeau. Le total de la colonne est donc toujours inférieur.
  • Libellé trompeur : « En attente attestation » compte en réalité « En attente de décision FEEF ».
  • Nemo Invest : la colonne affiche la date planifiée, vide pour cet audit. La vraie information serait la date de réalisation, non affichée.
À trancher avec la FEEF : pour un audit en attente d'avis, veut-on voir la date de réalisation ou la date planifiée ? Cela détermine si on change la colonne, ou si on répare la donnée.

Entités

« Total entités labellisées = 324 »

Ce libellé n'existe pas dans l'application. Le compteur s'appelle « Nombre d'entités ». C'est un comptage brut de la table, sans aucun filtre : il inclut les groupes, les suiveuses, les entités jamais auditées et même les entités supprimées.

Définition retenue le 08/07 : une entité est labellisée si elle est maître et qu'elle possède au moins un audit terminé. « Au moins un », et non « le dernier » : une entité en cours de renouvellement reste labellisée. La décision FEEF n'est pas utilisable comme critère, elle n'est pas fiable sur les audits importés.

Ce qui manque dans la liste

  • Aucune colonne parent ni enfants, alors que l'API charge déjà le parent et ne l'affiche jamais. Voir les rattachements sans faire du fiche à fiche est donc impossible aujourd'hui.
  • Un seul tri exposé (« dernière mise à jour ») alors que l'API en supporte huit.
  • « Dernier audit » est un badge sans date, et « dernier » signifie le plus récemment créé, pas le plus récent en date d'audit.
  • Pas de champ « statut » sur une entité. Le seul statut est celui du dernier audit.

Espace entreprise

Date de référence

Non modifiable : c'est volontaire, elle est réservée à la FEEF. Mais surtout, elle est vide pour toutes les entités importées : elle n'est posée qu'à la clôture d'un audit initial via le workflow, et les scripts d'import ne la remplissent jamais.

Conséquence à ne pas rater. La tâche automatique de relance d'anniversaire ignore toute entité sans date de référence. Les relances ne partiront jamais pour les entités importées.

« Comment je peux rattacher des entreprises ? »

Elle ne peut pas. Rattacher une entité existante est réservé à la FEEF (erreur 403). Une entreprise peut seulement créer une nouvelle suiveuse, jamais en rattacher une existante.

« Dans la frise étapes on n'a pas la liste des actions »

Conforme au code : la frise n'affiche que des statuts, à partir d'une liste figée. Elle ne consulte jamais la table des actions. Les actions s'affichent ailleurs dans l'application.

De plus, le script qui génère les actions ne le fait que pour les audits non terminés. LSDH n'a donc aucune action sur ses 8 audits terminés.

À confirmer : le script de génération des actions a-t-il été lancé en production ? Il n'est enchaîné à aucun autre et doit être exécuté explicitement, en dernier.

Besoin nouveau : permettre à la FEEF de corriger un audit

La FEEF doit pouvoir modifier le statut d'un audit et renseigner ses dates importantes, via un modal ou un panneau dédié. C'est ce qui permettra de traiter les 3 audits fantômes de 2022 sans intervention en base.

Option 1 | Transition forcée

Un endpoint qui fait avancer l'audit dans le workflow en court-circuitant les règles de validation.

  • Reste dans le modèle existant.
  • Les effets attendus continuent de s'exécuter : création des tâches, génération d'attestation.
  • Contourner les règles est précisément ce contre quoi le workflow protège.
  • Risque d'amener un audit dans un état incohérent, par exemple « en attente d'avis » sans rapport.
  • Effets de bord indésirables sur des données historiques : emails, notifications, attestations.
Recommandée

Option 2 | Édition directe, réservée à la FEEF

Un formulaire éditant le statut et les dates, hors workflow, avec journalisation de chaque modification.

  • Correspond au besoin réel : réparer des données d'import, pas faire avancer un dossier.
  • Aucun effet de bord. Entièrement traçable.
  • Deux chemins d'écriture coexistent sur le statut d'un audit.
  • Il faut garantir qu'aucune action ni notification n'est déclenchée.
